# Document Management Service

Тестовое задание --- backend‑сервис по работе с документами.

Документы создаются, переводятся по статусам, по изменениям статуса
ведётся история.\
Дополнительно реализована утилита (сервис) для массового создания
документов и фоновая обработка документов пачками.

------------------------------------------------------------------------

## Структура проекта

    document-management-service
     ├── init        # SQL-инициализация базы
     ├── management  # Основной сервис управления документами
     ├── generator   # Утилита массовой генерации и фоновой обработки
     └── common      # DTO и enums

------------------------------------------------------------------------

## Основной сервис: `management`

Функциональность:

-   Создание документа в статусе **DRAFT** (номер генерируется
    автоматически)
-   Получение одного документа вместе с историей изменений
-   Получение списка документов по списку ID (пакетное получение) с
    пагинацией и сортировкой
-   Отправка на согласование списка ID (1--1000):\
    `DRAFT → SUBMITTED`
-   Отправка на утверждение списка ID (1--1000):\
    `SUBMITTED → APPROVED`
-   Поиск документов по фильтру c пагинацией и сортировкой:
    -   статус
    -   автор
    -   дата создания
-   Конкурентное утверждение: запуск нескольких параллельных попыток
    утвердить один и тот же документ

------------------------------------------------------------------------

## Утилита и фоновая обработка: `generator`

При запуске:

-   Выполняется генерация документов со статусом **DRAFT** через API
    сервиса `management`

-   Размер пакета генерации задаётся в `application.yaml`:

    ``` yaml
    generating-size: 10000  # по умолчанию
    ```

Фоновые процессы:

-   `submitWorker` --- регулярно отправляет документы DRAFT → SUBMITTED через API
    сервиса `management`

-   `approveWorker` --- регулярно отправляет документы SUBMITTED →
    APPROVED через API сервиса `management`

-   Размер пачки задаётся в `application.yaml`:

    ``` yaml
    batch-size: 1000  # по умолчанию
    ```

Все задачи логируют ход выполнения в консоль.

## Запуск проекта

Проект --- Maven multi-module.

Для подключения библиотеки `common` и генерации мапперов необходимо в
родительском модуле выполнить:

    mvn clean install

### 1. Запуск PostgreSQL

В родительском модуле:

    docker compose up

### 2. Запуск основного сервиса

В модуле `management`:

    mvn spring-boot:run

### 3. Запуск генератора

В модуле `generator`:

    mvn spring-boot:run

## Предложения по доработке

_Что бы вы поменяли, чтобы обработка одного запроса уверенно работала с 5000+ id_
 - Отказаться от одиночных запросов и использовать пакетный или по критериям запрос в базу данных для изменения статуса. 
 - Асинхронная обработка, не дожидаться ответа, что бы канал не был занят продолжительное время между сервисами.
 - Можно записывать в отдельную таблицу список id для изменения статуса, а база данных через свою job сама изменит статус документа.

 _Как бы вы вынесли реестр утверждений в отдельную систему (отдельная БД или отдельный HTTP-сервис)_
 - Если нужно вынести реестр утверждений из-за нагрузки на базу данных, то отдельная база данных более предпочтительна, Spring Boot поддерживает Multitenancy. 
 - Можно исключить сервис и другая база данных через свою job основываясь на таблице с историей статусов будет вести реестр утверждений.
